# 参考 Starward 的功能扩展路线图

## 1. 参考资料

- 参考仓库：Starward
- GitHub 仓库链接：<https://github.com/Scighost/Starward>
- README 原文链接：<https://raw.githubusercontent.com/Scighost/Starward/main/README.md>

说明：
当前清单参考的是 Starward 在“米哈游账号资产聚合与个人使用体验”上的产品方向，而不是直接复制其桌面启动器形态。  
本项目现阶段仍以纯 Web/服务端架构为边界，优先选择能够复用现有账号、任务、日志、通知链路的能力。

## 2. 当前系统现状

当前仓库已经具备以下基础能力：

- 米游社账号扫码登录
- Cookie 与 `stoken` 登录态维护
- 手动签到与定时签到
- 签到日志查看与日历统计
- 邮件通知
- 登录态异常邮件提醒
- 管理员用户、系统 SMTP 与菜单显隐配置
- 抽卡记录中心 v1
  - 支持原神、星穹铁道完整链接导入
  - 支持 JSON 导入与 JSON 导出
  - 支持按账号与游戏重置抽卡记录及导入历史
  - 支持基础统计、卡池分布与明细分页查询

这意味着当前系统已经不再只是“自动签到工具”，而是已经具备“账号接入、定时任务、结果留痕、异常通知、资产归档”的基础底座。后续新增功能应优先复用这些已有能力，而不是另起一套独立流程。

## 3. 产品方向结论

基于当前系统能力边界与参考仓库定位，后续产品方向默认采用以下原则：

- 面向个人/小团体自托管用户
- 保持纯 Web/服务端架构，不主动引入桌面客户端依赖
- 从“自动签到平台”逐步升级为“米游社账号资产中心”
- 先形成完整路线图，再按优先级逐步实现

## 4. 功能分期

### P1 优先实现

#### 4.1 抽卡记录中心（已实现 v1）

- 参考来源：Starward 的抽卡记录保存能力
- 目标价值：
  让用户在同一套账号体系内长期保存抽卡记录，而不是依赖临时导出或第三方站点短期缓存。
- 与现有系统的复用关系：
  复用当前账号登录态、用户体系、日志能力与页面导航结构。
- 已实现范围：
  - 按账号选择支持游戏并导入抽卡记录
  - 原神、星穹铁道完整链接导入
  - JSON 导入与 JSON 导出
  - 数据去重与长期保存
  - 基础统计视图
  - 卡池分布展示
  - 按卡池筛选与分页查看明细
  - 按账号与游戏重置抽卡记录及导入历史
- 当前不纳入范围：
  - 高复杂度概率分析
  - 社区分享页
  - 跨平台公开查询接口
  - 更广游戏支持（当前仅支持 `genshin` / `starrail`）

#### 4.2 实时便笺面板（已实现 v1）

- 参考来源：Starward 这类账号中枢产品对“日常状态可视化”的价值方向
- 目标价值：
  让用户打开首页即可看到树脂、体力、派遣等关键状态，提升系统的日常打开频率，而不是只在签到失败时才使用。
- 与现有系统的复用关系：
  复用当前账号列表、登录态维护与首页仪表盘入口。
- 已实现范围：
  - 展示支持游戏账号的实时便笺摘要与角色卡片
  - 在首页仪表盘聚合展示实时便笺入口与状态
  - 标记异常卡片与空状态，引导用户前往账号接入
- 暂不纳入范围：
  - 高频推送提醒
  - 复杂自定义阈值规则

#### 4.3 兑换码中心（已实现 v1）

- 参考来源：个人账号工具常见的“集中兑换”场景，可与现有账号平台形成明显协同
- 目标价值：
  将“看到兑换码”和“批量给多个账号执行兑换”整合到系统内，减少手工逐个处理的成本。
- 与现有系统的复用关系：
  复用当前账号管理、任务执行、日志记录与通知链路。
- 已实现范围：
  - 录入兑换码并按游戏筛选可执行账号
  - 选择目标账号批量执行
  - 记录成功、失败、重复兑换、无效 Cookie 等结果
  - 支持按游戏查看历史批次与执行明细
- 暂不纳入范围：
  - 自动抓取互联网兑换码
  - 面向外部用户的开放提交入口

### P2 中期增强

#### 4.4 角色资产总览（已实现 v1）

- 参考来源：Starward 的个人账号聚合体验
- 目标价值：
  将当前“只按签到角色展示”升级为“按账号聚合角色资产”，为抽卡、便笺、战绩等后续能力提供统一入口。
- 与现有系统的复用关系：
  复用现有账号与角色表结构，并扩展角色视图语义。
- 已实现范围：
  - 按账号展示角色列表
  - 展示游戏、服务器、昵称、等级等基础资料
  - 聚合抽卡、便笺、兑换与签到入口，作为资产中枢页面
- 暂不纳入范围：
  - 深度角色养成分析
  - 构筑推荐

#### 4.5 账号健康中心（已实现 v1）

- 参考来源：将当前已有的登录态维护、签到结果、异常信息整合成统一运维面板
- 目标价值：
  让用户快速识别哪个账号能继续自动运行、哪个账号需要重新扫码、哪个账号近期失败频繁，减少排障成本。
- 与现有系统的复用关系：
  直接复用当前 Cookie 状态、自动续期状态、签到日志与通知记录。
- 已实现范围：
  - 展示 Cookie 状态
  - 展示自动续期可用性
  - 展示最近签到成功率与失败原因
  - 提示需要重新扫码、预警账号与健康账号数量
- 暂不纳入范围：
  - 企业级监控告警平台联动
  - 多租户 SLA 看板

#### 4.6 通知中心升级（已部分实现）

- 参考来源：当前系统已经具备邮件通知能力，但通知主题仍以签到为主
- 目标价值：
  将通知能力从“签到结果通知”升级为“账号资产中心统一通知”，避免后续每加一个功能就再造一套通知逻辑。
- 与现有系统的复用关系：
  复用现有 SMTP、用户通知邮箱、通知策略与通知服务。
- 已实现范围：
  - 登录态异常通知
  - `reauth_required` 单次告警去重
- 剩余主要实现范围：
  - 扩展到兑换码执行结果通知
  - 扩展到便笺异常或关键状态提醒
  - 在设置页中统一管理通知偏好
- 暂不纳入范围：
  - 短信、企业微信、钉钉等多渠道通知
  - 复杂通知编排

### P3 远期可选

#### 4.7 截图归档

- 参考来源：Starward 的截图查看能力
- 目标价值：
  作为个人游戏资产中心的补充能力，增强“个人账号资料库”的完整度。
- 与现有系统的复用关系：
  与当前 Web 架构复用度有限，需要额外的本地文件采集或上传能力。
- 主要实现范围：
  - 如未来允许本地代理或轻量客户端，再考虑截图采集与归档
  - Web 端只负责展示、搜索与整理
- 暂不纳入范围：
  - 当前阶段直接实现
- 延后原因：
  该能力依赖本地文件访问，和当前纯 Web/服务端边界不连续，优先级应低于资产聚合能力。

#### 4.8 游戏时长统计

- 参考来源：Starward 的游戏时长记录能力
- 目标价值：
  为用户提供更完整的个人使用画像，但不是当前系统的核心高频需求。
- 与现有系统的复用关系：
  几乎无法直接复用当前服务端能力，通常需要本地进程感知或启动器联动。
- 主要实现范围：
  - 仅在未来接受本地伴侣方案时再评估
- 暂不纳入范围：
  - 当前阶段直接实现
- 延后原因：
  对现有系统价值不如抽卡、便笺、兑换码直接，且实现成本明显更高。

#### 4.9 本地伴侣 / 桌面联动

- 参考来源：Starward 的桌面工具属性
- 目标价值：
  若未来产品形态扩展，可作为 Web 平台的补充入口，而不是当前主线。
- 与现有系统的复用关系：
  需要额外设计本地代理、客户端分发、版本兼容与安全策略。
- 主要实现范围：
  - 仅作为长期演进方向保留
- 暂不纳入范围：
  - 当前版本开发
- 延后原因：
  当前系统的竞争力仍在账号管理与自动化运维，而不是桌面启动器能力。

## 5. 实现原则

- 优先复用现有账号、任务、日志、通知链路，避免为新功能复制一套平行机制。
- 不为对齐参考仓库而强行引入与当前架构不连续的桌面能力。
- 每个功能进入开发前，仍需单独补充设计文档与实现计划，明确数据模型、接口边界、错误处理与验证方式。
- 优先实现能显著提升用户日常使用价值的能力，再考虑展示型或依赖本地环境的附加功能。

## 6. 后续执行建议

建议按以下顺序拆解后续工作：

1. 先推进 `通知中心升级`，把签到、登录态、兑换码、便笺异常统一到同一套通知偏好与模板语义
2. 再增强 `角色资产总览` 与 `实时便笺面板` 的联动深度，例如更多可视化聚合与跨模块跳转
3. 最后再评估 `截图归档`、`游戏时长统计`、`本地伴侣 / 桌面联动` 这类与当前纯 Web 架构边界更远的能力

这样可以在抽卡、便笺、兑换码、资产总览与健康中心都已具备首版落地的基础上，优先补齐“统一通知”和“跨模块资产聚合”两条主线，再逐步扩展外围能力。
